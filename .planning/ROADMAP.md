# Roadmap: Atrium

## Overview

Atrium is built in 8 tightly-scoped phases that mirror the architecture's dependency chain. The foundation and Canvas integration come first because nothing can run without data. A Todo List view validates that data in the browser before the schedule engine is built. Time estimates feed the scheduler. The scheduler's output is rendered in the calendar. Fixed blocking and preferences make the scheduler configurable and accurate. Each phase delivers something you can open and use — no phase ends with invisible plumbing.

## Phases

**Phase Numbering:**
- Integer phases (1, 2, 3): Planned milestone work
- Decimal phases (2.1, 2.2): Urgent insertions (marked with INSERTED)

Decimal phases appear between their surrounding integers in numeric order.

- [ ] **Phase 2: Canvas Sync** - Connect to Canvas API, pull all assignments with pagination and due-date overrides
- [ ] **Phase 3: Todo List** - Browser-visible assignment list with sync status and manual task creation
- [ ] **Phase 4: Time Estimator** - LLM-powered duration estimates with override and confidence display
- [ ] **Phase 5: Schedule Engine** - Greedy deadline-aware auto-scheduler with deadline conflict warnings
- [ ] **Phase 6: Calendar UI** - FullCalendar time-grid block view with drag-and-drop and click-to-complete
- [ ] **Phase 7: Fixed Time Blocking** - User-defined fixed events (classes, meetings) that the scheduler works around
- [ ] **Phase 8: Schedule Preferences** - Work window configuration wired into schedule regeneration

## Phase Details

### Phase 1: Foundation
**Goal**: The project builds and runs end-to-end with all data models defined and the TanStack Start / ElysiaJS / SQLite stack wired together
**Depends on**: Nothing (first phase)
**Requirements**: None (infrastructure phase — all 26 v1 requirements depend on this foundation)
**Success Criteria** (what must be TRUE):
  1. Running `bun dev` starts both the TanStack Start frontend and ElysiaJS backend with no errors
  2. All SQLite tables exist (assignments, time_estimates, schedule_blocks, fixed_events, preferences, completion_history) with WAL mode enabled
  3. A test API call from the browser reaches ElysiaJS via the Eden Treaty typed client and returns data
  4. Canvas API token is read from `.env` server-side and never exposed to the browser
**Plans**: 3 plans
Plans:
- [ ] 01-01-PLAN.md — Monorepo scaffold + @atrium/db full 6-table Drizzle schema with WAL mode
- [ ] 01-02-PLAN.md — ElysiaJS API server with CORS, health route, and Eden Treaty type export
- [ ] 01-03-PLAN.md — TanStack Start frontend with Eden Treaty client and end-to-end verification

### Phase 2: Canvas Sync
**Goal**: Users can connect their Canvas account and have all assignments pulled into the app automatically
**Depends on**: Phase 1
**Requirements**: CANV-01, CANV-02, CANV-03, CANV-04, CANV-05
**Success Criteria** (what must be TRUE):
  1. User can enter their Canvas API token and institution URL and the app connects successfully
  2. All assignments across all active courses appear in the database after sync, including assignments beyond the first page of results
  3. Each assignment shows the correct due date for the user's section (not just the default course deadline)
  4. The sync status badge shows which assignments are new, updated, or unchanged since the last sync
  5. User can press a "Sync Now" button to pull the latest Canvas data at any time

### Phase 4: Time Estimator
**Goal**: Every assignment has a duration estimate generated by LLM inference, which users can override, with a visible confidence indicator
**Depends on**: Phase 3
**Requirements**: EST-01, EST-02, EST-03, EST-04
**Success Criteria** (what must be TRUE):
  1. After sync, each assignment shows an AI-generated duration estimate (e.g., "2h 30m") without any manual input
  2. User can click an estimate and type a different duration to override it
  3. Each estimate displays a confidence badge or label showing its source (AI inference, user override, or historical data)
  4. The app tracks actual time spent on assignments and uses it to adjust future estimates for similar work

### Phase 5: Schedule Engine
**Goal**: The app automatically produces a time-blocked daily schedule from all incomplete assignments that is realistically usable and warns when deadlines are at risk
**Depends on**: Phase 4
**Requirements**: SCHED-01, SCHED-02, SCHED-03, SCHED-04
**Success Criteria** (what must be TRUE):
  1. After triggering generation, every incomplete assignment with a duration estimate appears as one or more scheduled work blocks
  2. The generated schedule includes gaps between blocks and is not perfectly packed wall-to-wall
  3. User can press "Regenerate Schedule" and get a fresh schedule reflecting current assignments and completion status
  4. If there is not enough time to complete all assignments before their deadlines, the app surfaces a visible warning naming the affected assignments

### Phase 7: Fixed Time Blocking
**Goal**: Users can define recurring and one-off fixed events so the scheduler never places assignment work on top of classes or meetings
**Depends on**: Phase 6
**Requirements**: BLOCK-01, BLOCK-02, BLOCK-03
**Success Criteria** (what must be TRUE):
  1. User can create a fixed block with a name, date, start time, and end time — and regenerating the schedule never places work inside that window
  2. User can create a recurring fixed block (e.g., every Tuesday 10am–12pm) that repeats each week
  3. User can edit the name, time, or recurrence of any fixed block, or delete it entirely
